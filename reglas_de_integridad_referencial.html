Trabajo Practico N°6

Reglas de Integridad Referencial

Definir las reglas de integridad referencial para las relaciones del proyecto "Liga de Estrellas". Se deberá detallar y justificar técnicamente la lógica aplicada en las operaciones de: Inserción - Modificación y Eliminación Para cada entidad relacionada, asegurando la consistencia de los datos según las reglas de negocio del torneo.

Justificación de Inserción (Regla: Restringir)

  • Temporadas - Grupos: No puedes crear un grupo si no existe una temporada activa donde ese grupo deba competir.
  • Temporadas - Fechas: Para insertar una "Fecha 1", el sistema debe verificar que exista la temporada, de lo contrario, la fecha no tendría un calendario de referencia.
  • Temporadas - Traspasos: Un pase de un jugador ocurre en un mercado de pases específico de una temporada. Si no hay temporada cargada, el traspaso no tiene contexto temporal.
  • Temporadas - Tabla de Posiciones: No se puede generar una tabla de puntos si no existe la temporada que la agrupa.
  • Temporadas - Temporadas_X_Torneos: Al ser una tabla de vinculación, necesita que la temporada exista primero para poder asociarla con los otros elementos (torneos, equipos, etc.).
  • Torneos - Temporadas: No puedes iniciar una "Temporada 2026" si no has definido a qué Torneo pertenece.
  • Torneos - Traspasos: Se debe restringir la inserción para asegurar que el traspaso se registre dentro de un torneo válido y oficial.
  • Torneos - Temporadas_X_Torneos: Para armar la estructura compleja de participación, el Torneo debe estar previamente creado en el sistema.
  • Grupos - Tabla de Posiciones: La tabla de posiciones se desprende de un grupo. Si intentas insertar una tabla sin grupo, el sistema no sabría qué equipos comparar.
  • Grupos - Temporadas_X_Torneos: El grupo es un componente de la estructura de juego; debe existir para poder asignar equipos a él.
  • Grupos - Entrenadores: No puedes asignar un entrenador a un grupo si el grupo no ha sido creado todavía.
  • Grupos - Partidos: Un partido debe pertenecer a un grupo (ej. Grupo A). Si el grupo no existe, el partido no tiene validez en la competencia.
  • Jugadores - Traspasos: Es imposible registrar un traspaso si el Jugador (el protagonista del pase) no existe en la base de datos.
  • Jugadores - Premios Jugadores: No se puede asignar un premio (como "Goleador") a un jugador que no ha sido dado de alta previamente.
  • Jugadores - Temporadas_X_Torneos: Para inscribir a alguien en la lista de buena fe del torneo, el Jugador debe estar registrado.
  • Jugadores - Planillas: Un jugador no puede aparecer en una planilla de partido si no existe en el sistema; esto evita el fraude de identidad.
  • Equipos - Traspasos: El traspaso requiere un equipo de origen o destino. Si el equipo no existe, el registro es inválido.
  • Equipos - Tabla de Posiciones: No puedes insertar un equipo en la tabla de puntos si el equipo no está en el maestro de clubes.
  • Equipos - Premios Equipos: Un premio (ej. "Valla menos vencida") solo puede asignarse a un equipo que figure en el sistema.
  • Equipos - Temporadas_X_Torneos: Al igual que los jugadores, los equipos deben estar registrados para participar en la estructura del torneo.
  • Equipos - Planilla: El equipo debe existir para ser seleccionado en el acta del partido.
  • Equipos - Partidos: No puedes programar un partido entre "Equipo A" y "Equipo B" si alguno de los dos no existe en la base de datos.
  • Premios Grupales - Premios Equipos: Para entregar un premio de tipo "Fair Play", primero debe existir la definición del premio en la tabla de "Premios Grupales".
  • Premios Individuales - Premios Jugadores: No puedes cargar que alguien ganó el "Botín de Oro" si el premio no está definido como una categoría existente.
  • Tipo de Torneos - Torneos: Un torneo debe ser de un tipo (ej. "Apertura" o "Clausura"). Si el tipo no existe, no puedes crear el torneo.
  • Categoría - Torneos: No se puede crear un torneo sin antes definir su categoría (ej. "Primera", "Veteranos").
  • Partidos - Planillas: La planilla es el registro de un partido. Si el partido no está programado, no se puede insertar su planilla.
  • Partidos - Resultados: No se puede cargar un "2 a 0" si el partido no existe previamente en el calendario.
  • Fechas - Partidos: Un partido se juega en una fecha determinada. El sistema restringe la inserción para asegurar que el partido se asocie a una jornada válida.

Justificación de Modificación (Regla: Cascada)

  • Temporadas - Grupos: Si se corrige el ID o código de una temporada, el grupo debe actualizar su referencia automáticamente para seguir perteneciendo a la misma.
  • Temporadas - Fechas: Las fechas están atadas cronológicamente a la temporada; si la temporada cambia su identificador, las fechas deben actualizarse para no quedar huérfanas.
  • Temporadas - Traspasos: El historial de pases debe reflejar siempre la temporada correcta. Si el código de la temporada se modifica, el registro del traspaso debe actualizarse para mantener la coherencia histórica.
  • Temporadas - Tabla de Posiciones: Si cambias el ID de la temporada, la tabla de posiciones debe actualizarse al instante para que los puntos sigan asignados al periodo correcto.
  • Temporadas - Temporadas_X_Torneos: Al ser una tabla relacional clave, cualquier cambio en la Temporada debe replicarse aquí para no romper la estructura de todo el torneo.
  • Torneos - Temporadas: Si el Torneo cambia de código, todas sus temporadas asociadas deben heredar ese cambio para no perder la jerarquía.
  • Torneos - Traspasos: La Cascada asegura que el traspaso siempre apunte al torneo correcto, incluso si los datos del torneo principal se editan.
  • Torneos - Temporadas_X_Torneos: Es vital que la tabla de asociación se actualice automáticamente ante cualquier cambio en el Torneo para mantener unidos a los equipos y jugadores.
  • Grupos - Tabla de Posiciones: Si el nombre o ID de un Grupo cambia, la tabla de posiciones debe reflejarlo para mostrar los resultados correctamente.
  • Grupos - Temporadas_X_Torneos: La asociación de equipos debe mantenerse firme; si el grupo se modifica, la tabla intermedia debe actualizar la referencia.
  • Grupos - Entrenadores: El entrenador debe seguir vinculado a su grupo aunque el identificador de dicho grupo sea modificado administrativamente.
  • Grupos - Partidos: Si el ID del grupo cambia, los partidos deben actualizar su referencia para que el sistema sepa a qué zona pertenecen esos puntos.
  • Jugadores - Traspasos: Si se corrige el DNI o el ID de un jugador, el registro de su pase debe actualizarse automáticamente para que el historial no se pierda.
  • Jugadores - Premios Jugadores: El premio debe seguir perteneciendo al mismo jugador aunque sus datos de identificación en el maestro sean modificados.
  • Jugadores - Temporadas_X_Torneos: La lista de buena fe debe actualizarse en cascada si hay cambios en los datos del jugador para que no haya errores de inscripción.
  • Jugadores - Planillas: Si el ID del jugador cambia, su presencia en las actas de partidos antiguos debe actualizarse para que las estadísticas sigan sumando a su perfil.
  • Equipos - Traspasos: Si un equipo cambia de código, los traspasos realizados deben reflejar el nuevo dato para mantener la legalidad del historial.
  • Equipos - Tabla de Posiciones: Es fundamental que si el equipo cambia de ID, la tabla de posiciones arrastre sus puntos al nuevo identificador.
  • Equipos - Premios Equipos: Los trofeos ganados deben seguir vinculados al club aunque este cambie su código en la base de datos.
  • Equipos - Temporadas_X_Torneos: La participación del club en el torneo depende de que esta tabla intermedia esté siempre sincronizada con el maestro de equipos.
  • Equipos - Planilla: Al modificar un equipo, las planillas de los partidos ya jugados deben actualizarse para que no aparezcan errores de "equipo no encontrado".
  • Equipos - Partidos: Si el ID del equipo cambia, el fixture debe actualizarse automáticamente para que los próximos partidos muestren los datos correctos.
  • Premios Grupales - Premios Equipos: Si se cambia el nombre de la categoría del premio, todos los equipos que lo ganaron deben ver el nombre actualizado.
  • Premios Individuales - Premios Jugadores: Si se edita la definición del premio individual, el cambio debe viajar a todos los jugadores que poseen dicho premio.
  • Tipo de Torneos - Torneos: Si el tipo de torneo se modifica, el torneo debe actualizar su referencia para mantener la regla de negocio.
  • Categoría - Torneos: Si se cambia el nombre o código de la categoría, el torneo debe reflejar este cambio en cascada.
  • Partidos - Planillas: Si el ID del partido se modifica, la planilla asociada debe actualizarse para no perder el acta del encuentro.
  • Partidos - Resultados: El resultado está pegado al partido; si el código del partido cambia, el marcador debe seguirlo mediante Cascada.
  • Fechas - Partidos: Si se cambia el ID de una fecha, todos los partidos programados en ella deben actualizar su referencia para aparecer en la jornada correcta.

Justificación de Reglas de Borrado

  • Temporadas - Grupos (Restringir): No puedes eliminar una temporada si ya tiene grupos configurados.
  • Temporadas - Fechas (Restringir): Las fechas dependen de la temporada. Se restringe para evitar perder el calendario programado accidentalmente.
  • Temporadas - Traspasos (Ignorar): El registro del traspaso puede quedar como historial sin afectar la integridad.
  • Temporadas - Tabla de Posiciones (Restringir): Borrar la temporada invalidaría las posiciones alcanzadas.
  • Temporadas - Temporadas_X_Torneos (Restringir): Borrar la temporada rompería toda la estructura jerárquica del torneo.
  • Torneos - Temporadas (Restringir): Si tiene temporadas activas o pasadas, el sistema impide su borrado.
  • Torneos - Traspasos (Ignorar): El traspaso es un dato de auditoría que puede mantenerse.
  • Torneos - Temporadas_X_Torneos (Restringir): Se bloquea el borrado porque esta tabla une la lógica de todo el sistema.
  • Grupos - Tabla de Posiciones (Restringir): La tabla depende directamente del grupo.
  • Grupos - Temporadas_X_Torneos (Restringir): Se restringe para no perder la organización de las zonas.
  • Grupos - Entrenadores (Restringir): No puedes borrar un grupo si tiene entrenadores asignados.
  • Grupos - Partidos (Restringir): No puedes borrar un grupo que ya tiene partidos programados o jugados.
  • Jugadores - Traspasos (Restringir): El historial de traspasos perdería sentido.
  • Jugadores - Premios Jugadores (Restringir): Se protege el mérito deportivo.
  • Jugadores - Temporadas_X_Torneos (Restringir): Si está inscripto en un torneo, el sistema impide su borrado.
  • Jugadores - Planillas (Restringir): Las planillas son actas oficiales.
  • Equipos - Traspasos (Restringir): No puede eliminarse para no romper el historial.
  • Equipos - Tabla de Posiciones (Restringir): No puedes borrar un equipo que figura en la tabla.
  • Equipos - Premios Equipos (Restringir): Se protege el palmarés del club.
  • Equipos - Temporadas_X_Torneos (Restringir): Si el equipo está asignado a una competencia, su borrado está bloqueado.
  • Equipos - Planilla (Restringir): Un equipo no puede desaparecer si ya fue asentado en una planilla.
  • Equipos - Partidos (Restringir): Evita inconsistencias en los partidos.
  • Premios Grupales - Premios Equipos (Restringir): No se puede borrar la categoría del premio si ya fue asignada.
  • Premios Individuales - Premios Jugadores (Restringir): No se borra la definición del premio si ya fue otorgado.
  • Tipo de Torneos - Torneos (Restringir): No puedes borrar un tipo de torneo mientras existan torneos usándolo.
  • Categoría - Torneos (Restringir): No puedes borrar la categoría si hay torneos asociados.
  • Partidos - Planillas (Restringir): Si el partido ya tiene planilla cargada, no puede borrarse.
  • Partidos - Resultados (Restringir): Protege la verdad deportiva.
  • Fechas - Partidos (Restringir): No puedes borrar una fecha si ya tiene partidos programados.
